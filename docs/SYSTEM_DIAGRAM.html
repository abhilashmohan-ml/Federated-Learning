System Overview

Top-level view of the five manufacturing sites, the central FL server, and the shared library. Each site runs an isolated Docker container. Only gradient updates (delta_W) travel between containers -- raw filtration data never leaves the site.

  • Shared Library -- physics models, schemas, crypto: used by both server and all clients
  • FL Server -- aggregates model updates, manages rounds, issues JWTs
  • Site Clients (x5) -- load data locally, train, add DP noise, upload delta_W

Federated Learning Protocol Flow

Sequence of calls across one complete FL round. Each round produces a new version of the global model. Sites participate independently -- no direct site-to-site communication ever occurs.

sequenceDiagram participant OP as Operator participant SRV as FL Server participant DB as Database participant SITE as Site FLClient participant TR as LocalTrainer Note over OP,DB: 1) Start a new FL round OP ->> SRV: POST /federation/round/start SRV ->> DB: INSERT rounds status=collecting, round_id=1 SRV ->> SRV: asyncio: spawn _timeout_guard task SRV -->> OP: FederationRound round_id=1 Note over SITE,TR: 2) Site detects round and downloads global model SITE ->> SRV: GET /federation/round/1 SRV -->> SITE: status=collecting SITE ->> SRV: GET /models/global-model SRV -->> SITE: global weights W_current Note over SITE,TR: 3) Local training (data stays on-site) SITE ->> TR: train_and_prepare_update(round_id=1) TR ->> TR: load_filtration_csv() TR ->> TR: fit_all_models() Hermia AIC selection TR ->> TR: compute_flux_ratio() compute_amin() TR ->> TR: add_gaussian_noise(delta_W, sigma) TR -->> SITE: ModelUpdate delta_W, n_samples, metrics Note over SITE,SRV: 4) Upload privacy-protected update SITE ->> SRV: POST /federation/update SRV ->> DB: INSERT model_updates SRV -->> SITE: 200 accepted Note over SRV,DB: 5) Aggregate after all sites submit SRV ->> SRV: FedProxAggregator.aggregate(updates, W_current) SRV ->> SRV: W_new = sum(n_i / N * (W_current + dW_i)) SRV ->> DB: UPDATE rounds SET status=complete SRV ->> SRV: store new global weights in ModelRegistry Note over SITE,SRV: 6) Next round begins SITE ->> SRV: GET /models/global-model SRV -->> SITE: GlobalModel version=2

Authentication and Token Flow

Complete lifecycle of JWT tokens -- from initial credential exchange through token rotation and revocation. All manufacturing sites follow this same flow independently. Access tokens expire after 15 minutes; refresh tokens are single-use with a 7-day lifetime.

sequenceDiagram participant SITE as FLClient participant AUTH as /auth API participant DB as Database participant PROT as Protected Routes Note over SITE,DB: 1) Initial Authentication SITE ->> AUTH: POST /auth/token site_id + site_secret AUTH ->> DB: SELECT site_registry WHERE site_id DB -->> AUTH: row with secret_hash AUTH ->> AUTH: bcrypt.verify(secret, hash) AUTH ->> AUTH: issue access token 15min with jti_A AUTH ->> AUTH: issue refresh token 7d with jti_R AUTH -->> SITE: access_token + refresh_token Note over SITE,PROT: 2) Using the access token SITE ->> PROT: POST /federation/update Bearer access_token PROT ->> PROT: jwt.decode(token, secret_key) PROT ->> PROT: expiry check valid PROT -->> SITE: 200 accepted Note over SITE,DB: 3) Token rotation (access expires after 15 min) SITE ->> AUTH: POST /auth/refresh refresh_token AUTH ->> AUTH: decode refresh token, extract jti_R AUTH ->> DB: SELECT revoked_tokens WHERE jti = jti_R DB -->> AUTH: NULL not revoked yet AUTH ->> DB: INSERT revoked_tokens jti_R consume single-use token AUTH ->> AUTH: issue new access and refresh pair AUTH -->> SITE: new access_token + new refresh_token Note over SITE,DB: 4) Replay attack blocked SITE ->> AUTH: POST /auth/refresh old refresh_token AUTH ->> DB: SELECT revoked_tokens WHERE jti = jti_R DB -->> AUTH: row found already consumed AUTH -->> SITE: 401 Unauthorized Token revoked

Docker Network Isolation

Network topology in the Docker Compose deployment. Each site is isolated on its own bridge network. Sites cannot communicate with each other or with the database directly.

  • db_net: server and PostgreSQL only -- no site container has access
  • site_N_net: server and site_N only, one per site (6 networks total)
  • site_1 cannot reach site_2, site_3, site_4, site_5, or PostgreSQL directly
  • All gradient updates flow through the server -- never site-to-site
